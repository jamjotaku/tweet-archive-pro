# TweetArchive Pro 追加機能の実装プラン (低負荷設計版)

ユーザーのPC（ブラウザ）のスペックにも、本番サーバー（Fly.ioの無料枠）のスペックにも過剰な負担をかけないための、バランスの取れた設計アプローチを採用します。

## ⚠️ User Review Required (負荷分散と設計方針)

> [!TIP]
> **「ユーザーのPCに依存しない」ための解決策**
> 
> *   **1. Markdown & コードハイライト**:
>     *   ブラウザで毎回変換するのではなく、**「保存時」にサーバー側で一度だけMarkdownをHTMLに変換**し、DBに保存します。（サーバー負荷は一瞬でごく僅かです）。表示時はDBに保存された完成済みのHTMLをそのまま表示するだけなので、どんなに古いスマホから見ても一切負荷がかかりません。
> *   **2. ビジュアル・ナレッジグラフ**:
>     *   数千件のツイートを一度に物理演算（力学モデル）で描画すると、ユーザーのPCがフリーズする原因になります。
>     *   そこで今回は、全体を一度に描画するのではなく、絞り込みを前提とした**「ローカルグラフ（関連表示）」**を採用します。特定のカテゴリやタグを選択した際、または特定のブックマークを開いた際に、**「そこから2〜3階層以内のみの繋がり（最大数十個）」を計算して描画**します。これにより、スマホや低スペックPCでもサクサク動く美しいグラフを実現できます。
> *   **3. AIによる自動タグ・要約**:
>     *   引き続き、外部のAPI（**Gemini API** など）をサーバー経由で呼び出すようにします。これによりユーザーPCにも、Fly.ioサーバーにも重いAI演算処理の負荷はかかりません。

この「事前変換」と「ローカルグラフ（部分描画）」のアプローチで堅牢に作っていく方針でよろしいでしょうか？

## Proposed Changes

### Phase 1: Markdown & コードハイライト対応 (事前パース方式)

#### [MODIFY] `app/models.py`
- `Bookmark` モデルに、変換済みのHTMLを保存するための `note_html` (Text) カラムを拡張します。

#### [MODIFY] `app/crud.py`
- ブックマーク追加・更新の際、Pythonの `markdown` パッケージを使用して `note` (Markdown本文) を `note_html` (HTML) に変換してからDBに保存します。

#### [MODIFY] `app/templates/index.html` & `app/templates/base.html`
- フロントエンドではJSによる重い変換処理を行わず、サーバーから送られてきた `note_html` を安全に（サニタイズして）表示するだけに変更します。（コードハイライト用の超軽量なCSSのみ追加します）。

---

### Phase 2: ビジュアル・ナレッジグラフ (ローカルグラフ方式)

#### [NEW] `app/routers/graph.py` (または main.py 追加)
- `GET /bookmarks/graph?focus_id=123` のように、**中心となるIDやタグを指定**するグラフ構造取得APIを作成します。
- サーバー側でSQLを駆使し、中心のノードから N回層 以内のデータ（最大 50〜100ノード程度）だけを抽出してJSON化します。

#### [NEW] `app/templates/graph.html`
- 軽量なグラフ描画ライブラリ（`vis-network` など）を組み込だ専用ビューワ。数百ノード以下に絞られているため、どんな端末でも60fpsでスムーズに動くUIを目指します。

---

### Phase 3: AIによる自動タグ付け＆要約

#### [MODIFY] `app/main.py`
- ブックマーク追加時、バックエンドで非同期タスクとしてGemini APIに本文を投げ、「ハッシュタグ」「カテゴリ」「3行要約」をJSONで取得します。（失敗時は元の本文のみを使用）。

#### [NEW] `app/ai_helper.py`
- `google-generativeai` ライブラリを用いたAPI通信クラスを実装し、エラーハンドリング（タイムアウトなど）を厳密に設定します。

---

## Validation Plan

1. **Markdown事前パース**: ローカルDBマイグレーション実行後、マークダウンを保存し、サーバー側の力で瞬時にHTMLでレンダリングされて画面に出るか確認。
2. **Local Graph**: 画面に「グラフを表示」ボタンを設置し、クリック時に周囲の関連カードだけがスムーズなアニメーションで描画され、ユーザーの端末（ブラウザ）のCPU使用率が跳ね上がらないか確認。
3. **AI Integration**: `.env` に仮のAPIキーをセットし、ツイートを保存した際にAIが的確なタグと要約を返してくるかを検証。
